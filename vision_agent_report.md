# Vision Agent 模块汇报

---

## 一、整体概览：输入与输出

Vision Agent 是整个平台的"眼睛"——负责把用户上传的**照片**变成**结构化数据**，供下游的 LLM 大脑、护理 Agent、预警 Agent 等模块使用。

### 一句话总结

> **照片进来 → 结构化 JSON 出去**

### 支持的三个场景

| 场景 | 用户上传什么 | 系统输出什么 |
|------|-------------|-------------|
| 饮食打卡 | 三餐照片（如海南鸡饭、Kopi） | 食物名称、份量、卡路里、碳水、蛋白质、脂肪等营养数据 |
| 药物识别 | 药盒、处方单、补剂瓶（支持多张，如正面+背面） | 药名、剂量、服用频次、给药方式、有效期、警告信息 |
| 医疗报告 | 纸质化验单、体检报告 | 所有检测指标（名称、数值、单位、参考范围、是否异常） |

**额外说明**：如果上传的图片不属于以上三类（比如自拍、风景），系统会自动拒识并告诉用户"这张图无法分析"。

## 二、技术架构：LangGraph 节点流程

整个 Vision Agent 使用 **LangGraph**（一个状态图编排框架）构建，数据在节点之间像流水线一样传递。

### 架构图

```
                        ┌─────────────────────────────────────────────┐
                        │            Vision Agent Pipeline            │
                        └─────────────────────────────────────────────┘

 用户上传照片（1~5张）
        │
        ▼
 ┌──────────────┐
 │ image_intake  │  图片预处理：验证格式/大小 → 转为 base64 编码
 └──────┬───────┘
        │
        ▼
 ┌──────────────────┐
 │ scene_classifier  │  场景分类：这是食物？药物？报告？还是无关图片？
 └──────┬───────────┘
        │
        ├── 食物 FOOD ──────────┐
        ├── 药物 MEDICATION ────┤
        ├── 报告 REPORT ────────┤
        └── 无关 UNKNOWN ───────┤
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
             ┌────────────┐ ┌──────────┐ ┌──────────────┐
             │food_analyzer│ │medication│ │report_       │  ... 或 rejection_handler
             │  饮食分析   │ │_reader   │ │digitizer     │      （拒识处理）
             │            │ │ 药物识别  │ │ 报告数字化    │
             └─────┬──────┘ └────┬─────┘ └──────┬───────┘
                   │             │              │
                   └─────────────┼──────────────┘
                                 │
                                 ▼
                       ┌─────────────────┐
                       │ health_advisor   │  健康建议：把识别结果交给文本 LLM
                       │ （SeaLION 大脑） │  生成本土化、针对慢病患者的建议
                       └────────┬────────┘
                                │
                                ▼
                       ┌─────────────────┐
                       │ output_formatter │  统一格式化 + Pydantic 数据校验
                       └────────┬────────┘
                                │
                                ▼
                         最终结构化输出
                     （JSON，供下游模块使用）
```

### 各节点详解

#### 1. image_intake — 图片预处理

接收用户上传的照片（支持 1~5 张），逐个检查：
- 文件格式是否支持（JPG/PNG/WebP/BMP/AVIF）
- 文件大小是否超过 10MB
- 全部通过后，将图片转为 base64 编码（VLM 模型需要的输入格式）

**为什么支持多张图片？** 实际场景中，药盒需要拍正面和背面才能看全信息；多页报告一张图拍不完。多图支持让识别信息更完整。

#### 2. scene_classifier — 场景分类

拿到图片后，第一步是判断"这张图属于什么场景"。VLM 会把图片分为四类之一：
- **FOOD** — 食物/饮料
- **MEDICATION** — 药物/处方/补剂
- **REPORT** — 化验单/体检报告
- **UNKNOWN** — 无关图片（自拍、风景等）

分类结果决定了接下来走哪个专门的分析节点。

**Prompt 设计要点**：
- 给 VLM 明确定义了每个类别包含什么（例如 MEDICATION 不仅包括药盒，还包括胰岛素笔、血糖仪、药房配药袋等）
- 要求输出 confidence（置信度），为下游判断提供依据
- 让 VLM 给出 reason（分类理由），便于调试

#### 3. food_analyzer — 饮食分析

识别照片中**所有可见的食物**，对每一项估算营养数据。

**输出字段**：
- `items` — 食物列表，每项包含：
  - `name`：食物名称（优先使用本地名，如 "Char Kway Teow" 而不是 "fried noodles"）
  - `quantity`：份量估算（如 "1 plate"、"250ml"）
  - `nutrition`：营养数据（calories 卡路里、carbs 碳水、protein 蛋白质、fat 脂肪、fiber 膳食纤维、sodium 钠）
- `total_calories_kcal` — 所有食物的总卡路里
- `meal_type` — 餐次（早餐/午餐/晚餐/宵夜/零食）

**Prompt 设计要点**：
- 内置 **50+ 新加坡本地菜名**参考列表（海南鸡饭、叻沙、肉骨茶、炒粿条、Nasi Lemak、Roti Prata、Kaya Toast 等），引导 VLM 使用精确的本地名称
- 内置**新加坡份量标准**（如小贩中心一份 350-500g，一碗汤面 300-450ml）
- 要求按 HPB（新加坡健康促进局）数据估算营养值
- 要求包含配菜、调味料和饮料（不只是主食）

#### 4. medication_reader — 药物识别

从药盒/处方单/补剂瓶中提取所有可见的药物信息。

**输出字段**：
- `drug_name`：药物全名（通用名 + 品牌名，如 "Metformin Hydrochloride (Glucophage)"）
- `dosage`：剂量（如 "500mg"、"200mg per 3 capsules"）
- `frequency`：服用频次（如 "twice daily with meals"）— 如果图片上没有标注则为 null
- `route`：给药方式（oral 口服 / injection 注射 / topical 外用）
- `warnings`：警告信息（原样提取）
- `expiry_date`：有效期

**Prompt 设计要点**：
- 内置**新加坡常见慢病药物清单**（糖尿病：Metformin、Gliclazide、Insulin Lantus 等；高血压：Amlodipine、Losartan 等；胆固醇：Atorvastatin 等）
- 识别新加坡特有的**频次缩写**（OD = 每天一次，BD = 每天两次，TDS = 每天三次，PRN = 按需）
- 了解新加坡 **Polyclinic 配药标签格式**（黄色/白色贴纸，四种语言标注）
- 识别 **HSA（卫生科学局）注册编号**

#### 5. report_digitizer — 报告数字化

从纸质化验单中提取**所有检测指标**。

**输出字段**：
- `report_type`：报告类型（血液检查/尿检/血脂/肝肾功能等）
- `indicators`：指标列表，每项包含：
  - `name`：指标名称（如 "HbA1c"、"Fasting Blood Glucose"）
  - `value`：数值（如 "7.2"、"120/80"）
  - `unit`：单位（如 "mmol/L"、"g/L"）
  - `reference_range`：参考范围
  - `is_abnormal`：是否异常（true/false）
- `report_date`：报告日期
- `lab_name`：医院/实验室名称

**Prompt 设计要点**：
- 内置**新加坡 MOH/HPB 标准参考范围**（如 HbA1c <5.7% 正常、≥6.5% 糖尿病；空腹血糖 3.9-6.0 mmol/L 正常）
- 使用**亚洲人体标准**（如 BMI 用亚洲切点：≥23 为超重，而不是西方的 ≥25）
- 覆盖新加坡主要医院格式（SGH、NUH、TTSH、CGH 及综合诊所）
- 要求提取报告上的**每一个**指标，不遗漏

#### 6. health_advisor — 健康建议（SeaLION 大脑）

这个节点不看图片，而是把前面的**识别结果**（结构化 JSON）交给 SeaLION 文本模型，生成针对新加坡慢病患者的建议。

**三个场景各自的建议内容**：

| 场景 | 建议内容 |
|------|---------|
| 饮食 | GI 影响评估、是否适合糖尿病患者、实用替换建议（如"teh tarik 换 teh-O kosong"） |
| 药物 | 药物用途简单解释、服药提醒、食物/药物相互作用 |
| 报告 | 异常指标解释、下一步行动建议、生活方式建议 |

**Prompt 设计要点**：
- 食物建议的语气是 **"像 kopitiam 里懂行的 auntie/uncle 给建议"**，用 Singlish，不说教
- 药物建议模拟 **Polyclinic 药剂师** 的角色
- 报告建议参考 **MOH/HPB 慢病管理指南**

#### 7. output_formatter — 输出格式化

最后一个节点。做两件事：
- 用 **Pydantic v2**（数据校验框架）验证输出的每个字段是否符合预定义的 schema
- 统一格式化，确保下游模块拿到的数据结构一致

## 三、双模型架构：Gemini（眼睛）+ SeaLION（大脑）

Vision Agent 使用两个模型协作：

```
  照片 ──→ Gemini 2.5 Flash ──→ 结构化识别结果 ──→ SeaLION 27B ──→ 本土化健康建议
           （视觉模型-眼睛）                         （文本模型-大脑）
           看图 → 识别内容                           理解结果 → 给建议
```

| | Gemini 2.5 Flash | SeaLION 27B |
|---|---|---|
| **角色** | 眼睛 — 看图识别 | 大脑 — 理解分析 |
| **输入** | 图片 + Prompt | 识别结果的 JSON 文本 |
| **输出** | 结构化 JSON（食物/药物/报告信息） | 健康建议 JSON |
| **为什么选它** | 视觉能力强，支持多图 | AI Singapore 本土模型，懂 Singlish、懂新加坡文化 |

**为什么不用一个模型做所有事？** 因为"看得准"和"说得好"是两种不同的能力。Gemini 视觉识别强但不了解新加坡本土文化；SeaLION 是新加坡本土训练的模型，更懂本地语境，但目前没有开放视觉 API。两个模型各发挥长处。

## 五、下一步计划

### 近期（Short-term）

1. **补剂多成分 Schema 优化** — 当前 schema 只支持单一药物，需扩展以支持补剂的多成分结构（如 Lutein 10mg + DHA 300mg + B12 5mg）
2. **收集更多测试图片** — 目标 20 张，覆盖更多场景（模糊图、手写处方、复杂背景食物）
3. **Prompt 迭代** — 根据真实图片测试结果持续优化
4. **FoodAI 接入** — 审批通过后，食物场景从 Gemini 切换为 FoodAI（SG 专精）

### 中期（Medium-term）

5. **多图部分失败恢复** — 多张图中如果某张有问题，跳过继续分析，而不是整体报错
6. **置信度校准** — 让 confidence 分数更有区分度，方便下游判断是否需要向用户确认

## 六、未来方向

### A. 视觉识别本身的优化

| 方向 | 说明 |
|------|------|
| **营养数据库接入** | VLM 负责"看"（识别菜名），外部营养数据库负责"查"（精确营养值），取代目前 VLM 自己估算的方式 |
| **RxNorm 药物验证** | VLM 提取药名后，用 RxNorm API（美国国家医学图书馆，免费）进行标准化验证 |
| **OCR 增强** | 对模糊/手写的处方单/报告，先跑 OCR 提取文字，再交给 VLM 分析 |

### B. 与其他组员模块的 Pipeline 对接方向

Vision Agent 的输出不仅是数据，**字段的缺失本身也是信号**。下游模块可以利用这些信号触发交互：

| 信号 | 下游可以做什么 |
|------|--------------|
| 药物的 `frequency` 为 null | LLM 大脑追问用户："这个补剂没有标注服用频次，你通常怎么吃？" |
| 药物的 `dosage` 为 null | LLM 大脑提醒："照片上看不清剂量，你能确认一下吗？" |
| 食物的 `quantity` 不确定 | LLM 大脑询问："这份看起来是鸡饭，你吃了多少？整份还是半份？" |
| `confidence` 较低（<0.5） | LLM 大脑不自动使用结果，而是主动询问用户确认 |
| 报告的某指标 `is_abnormal` = true | 预警 Agent 触发健康预警 |

**追问循环（未来可做）**：
```
照片 → Vision Agent 识别（发现缺失字段）
  → 下游 LLM 追问用户补充信息
  → 用户补充后回填到结构化数据
  → 或者用户再拍一张图补充 → 再次分析
```
